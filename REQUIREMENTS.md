# TrackFolio 需求文档

## 1. 文档信息

- 项目名称：TrackFolio
- 产品定位：单人自托管的股票与基金盈亏盯盘分析网站
- 文档版本：v1.4
- 编写日期：2026-06-06
- 最近修订：2026-06-11
- 当前阶段：MVP 已实现，需求持续维护
- 修订说明：v1.4 对齐当前项目实现：补充资产配置 JSON 导入导出；导入通过创建资产和买入交易还原当前活跃持仓。完整交易流水备份恢复、CSV 导出、列显隐、价格提醒等仍为后续迭代。

## 2. 产品概述

TrackFolio 面向个人投资者和主动盯盘用户，用于集中管理 A 股、美股、港股股票、ETF、场内基金和场外基金持仓，实时查看今日盈亏、昨日盈亏、历史盈亏和总持仓盈亏。

产品首屏直接进入盈亏看板，不做营销落地页。界面风格偏极客、干净、图表化，强调数据密度、刷新状态、跨币种汇总和快速定位异常波动。

TrackFolio 当前是单人自托管应用：前端页面与后端 API 由同一个 Fastify 服务同源托管，默认使用 SQLite，支持切换 PostgreSQL。

## 3. 目标与边界

### 3.1 产品目标

- 支持用户手动添加或搜索添加 A 股、美股、港股股票和基金持仓。
- 通过买入 / 卖出交易流水自动推算持仓数量、平均成本、累计费用和清仓状态。
- 展示今日盈亏、昨日盈亏、历史盈亏、总持仓盈亏和持仓市值。
- 支持 CNY、USD、HKD 统一结算货币，用于跨市场、跨币种资产的盈亏汇总展示。
- 支持配置结算时区，保证历史快照和图表日期口径稳定。
- 支持实时行情 / 基金净值刷新，并明确展示价格更新时间、市场状态、刷新状态和数据缺口。
- 支持定投批量补录；净值未披露的期次允许暂存为待确认，待后台获取净值后自动折算为正式交易流水。
- 支持资产配置 JSON 导入导出，用于迁移或快速恢复当前活跃持仓配置。
- 通过图表、高密度表格和移动端卡片帮助用户快速判断账户盈亏来源。

### 3.2 MVP 不包含

- 不包含自动交易、自动下单、自动调仓。
- 不包含资金托管、出入金、支付能力。
- 不包含券商账户自动同步。
- 不支持多用户、登录注册、角色权限和团队协作。
- 不提供投资建议、荐股、预测收益或自动买卖信号。
- 不要求绑定单一行情供应商；行情、基金净值和汇率服务应通过可替换 Provider Adapter 接入。

### 3.3 当前实现边界

当前已实现：

- 首页盈亏看板、总览卡、持仓列表、历史曲线和盈亏贡献图。
- 后台设置页，包含后台密码解锁、资产配置、资产导入导出、交易编辑、清仓、删除、显示设置、行情校验、汇率刷新和密码修改。
- A 股 / 美股 / 港股股票与基金资产维护。
- 买入 / 卖出交易流水、批量定投补录、待确认净值回填。
- CNY / USD / HKD 统一结算和汇率 Provider。
- 深色 / 浅色 / 自动 / 自定义主题、背景图、涨跌配色、刷新间隔、结算时区。
- 移动端基础布局，包括持仓卡片、后台移动端排序与弹窗适配。

当前未完整实现，作为后续迭代：

- 独立资产详情页与单资产完整历史趋势页。
- 持仓列表列显隐和 CSV 导出。
- 完整交易流水、设置与历史数据的备份与恢复。
- 按市场 / 资产类型拆分的完整历史贡献图。
- 分红、拆股、配股、税费等复杂交易事件。
- 风险敞口、行业集中度、自定义看板、价格提醒、盈亏预警和 PWA。

## 4. 目标用户与核心场景

### 4.1 目标用户

- 同时持有 A 股、美股、港股股票、ETF 和基金的个人投资者。
- 需要在交易日快速查看持仓盈亏变化的盯盘用户。
- 习惯用手动方式维护持仓和交易记录的用户。
- 有定投记录补录需求、关注基金净值确认口径的用户。
- 关注数据准确性、可视化分析、移动端可查看和简洁界面的极客型用户。

### 4.2 核心场景

- 用户打开网站，立即看到整体资产今日盈亏、昨日盈亏、累计盈亏、总持仓盈亏和持仓市值。
- 用户切换统一结算货币，将 A 股、美股、港股和基金盈亏统一折算为 CNY、USD 或 HKD 展示。
- 用户进入后台解锁后，添加一只股票或基金，并录入首笔买入交易，系统据此建立持仓并自动计算数量、平均成本和累计费用。
- 用户在后台导出当前活跃持仓配置，或导入此前导出的 JSON 文件快速恢复资产配置。
- 用户补录基金定投记录，一次生成多期买入交易；若某期净值未披露，系统先保存待确认记录，后台定期尝试回填。
- 用户按市场、资产类型、搜索关键字或盈亏排序查看持仓明细。
- 用户查看近 7 / 30 / 90 天、今年或自定义区间的历史盈亏曲线。
- 行情、基金净值或汇率服务异常时，用户能看到最后成功更新时间、受影响资产和当前数据状态。
- 用户在后台调整主题、背景图、涨跌配色、刷新间隔、结算货币和结算时区。

## 5. 功能需求

### 5.1 总览看板

首页直接展示资产总览，包含以下核心指标：

- 总资产市值：当前持仓按最新价格或最新净值折算后的总市值。
- 今日盈亏：基于最新实时价、最新净值、上一交易日收盘价、上一净值和当日交易计算的当日持仓盈亏。
- 昨日盈亏：基于昨日收盘价、昨日净值与前一交易日收盘价或前一净值计算的昨日持仓盈亏。
- 历史累计盈亏：基于 DailyPnL 快照展示历史累计盈亏走势。
- 总持仓盈亏：基于当前价格或最新净值、成本价、持仓数量和费用计算。

总览看板必须显示：

- 当前行情更新时间。
- 持仓涉及市场的交易状态，例如盘中、盘前、盘后、休市、数据延迟。
- 数据刷新状态，例如刷新中、成功、失败、部分失败。
- 当前统一结算货币。
- 汇率异常或部分资产不可折算时的提示。
- 盈亏金额和盈亏比例，正负值使用不同颜色区分。
- 汇总指标默认按统一结算货币展示，持仓明细可展示原币金额和折算金额。

### 5.2 资产添加

系统支持添加 A 股、美股、港股股票和基金。

支持资产类型：

- 股票：A 股、美股、港股。
- 场内基金：A 股 ETF、LOF、REITs，美股 ETF，港股 ETF。
- 场外基金：以基金代码和单位净值维护的公募基金或自定义基金。

实现口径：资产主类型为 `STOCK` 或 `FUND`；基金细分通过 `fund_type` 表达，例如 `etf`、`otc` 等。

统一资产标识格式：

- A 股股票：`STOCK:CN:600519`
- 美股股票：`STOCK:US:AAPL`
- 港股股票：`STOCK:HK:00700`
- A 股 ETF：`FUND:CN:510300`
- 场外基金：`FUND:CN:000001`

添加方式：

- 搜索添加：用户输入代码、名称或拼音缩写后选择股票或基金。
- 手动添加：用户选择资产类型、市场并输入代码、名称、币种。
- 自定义标的：行情或净值服务无法识别时，允许在用户确认后保存为自定义标的。

添加校验：

- 单实例内 `asset_type + market + symbol` 不允许重复创建为不同资产。
- 资产类型必须为 `STOCK` 或 `FUND`。
- 市场必须为 `CN`、`US`、`HK` 之一。
- A 股 / A 股基金代码默认 6 位数字；港股代码默认 4-5 位数字；美股代码默认 1-6 位大写字母，可带点后缀。
- 币种默认值：A 股为 CNY，美股为 USD，港股为 HKD。
- 场外基金默认币种为 CNY，用户可手动调整。
- 手动添加资产时，如行情或净值服务无法识别，应允许保存为自定义标的，但必须标记为“数据不可用”或“待刷新”。

### 5.3 持仓管理

持仓以**交易流水驱动**：用户录入买入 / 卖出交易，系统按加权平均法自动推算持仓数量与平均成本，持仓数量和平均成本不允许手动直接修改。

用户可维护或查看的持仓信息：

- 资产类型
- 市场
- 资产代码
- 资产名称
- 持仓数量或基金份额（由交易流水计算，只读）
- 平均成本价（按加权平均从交易流水自动计算，只读）
- 累计交易费用（由交易流水累计，只读）
- 币种
- 建仓日期（取首笔买入交易时间，可在后台调整元数据）
- 标签（持仓级元数据，可编辑）
- 备注（持仓级元数据，可编辑）
- 清仓时间（全部卖出后自动标记）

成本计算规则：

- 买入：新持仓数量 = 原数量 + 买入数量；新平均成本 = (原平均成本 × 原数量 + 买入价 × 买入数量) / 新持仓数量。
- 卖出：持仓数量减少，平均成本价不变（加权平均法）。
- 交易费用单独累计，不计入平均成本价，在总持仓盈亏中单列扣除。
- 全部卖出后持仓数量归零并标记清仓，交易流水保留以供历史追溯。
- 删除或编辑交易后，系统必须自动重算对应持仓、清仓状态和相关历史数据。

持仓操作：

- 录入买入交易（建仓 / 加仓）。
- 录入卖出交易（减仓 / 清仓）。
- 批量录入买入交易（定投补录）。
- 编辑持仓元数据（标签、备注、建仓日期）。
- 编辑交易。
- 删除交易（删除后自动重算持仓）。
- 清仓持仓。
- 删除持仓或资产。
- 查看该资产交易流水。

录入交易、删除交易、编辑交易、删除资产、清仓等写操作必须在后台解锁状态下执行。后台使用单实例管理密码保护，默认密码为 `admin`，进入后台后应立即修改。删除资产时必须二次确认；已清仓或已产生历史记录的持仓建议软删除或归档，避免历史盈亏曲线断裂。

### 5.4 资产配置导入导出

系统支持在后台导出和导入当前活跃持仓配置，用于迁移或快速恢复资产配置。

导出要求：

- 后台解锁后可点击“导出资产”。
- 导出格式为版本化 JSON，schema 固定为 `trackfolio.assetAllocation.v1`。
- 导出内容包含当前活跃持仓的资产标识、名称、市场、币种、数量、平均成本、累计费用、建仓时间、标签和备注。
- 导出文件命名建议为 `trackfolio-allocation-YYYY-MM-DD.json`。
- 导出文件不包含完整交易流水、定投期次、待确认净值、历史行情、显示设置、后台密码或数据库内部状态。

导入要求：

- 后台解锁后可点击“导入资产”选择 JSON 文件。
- 导入只接受 `trackfolio.assetAllocation.v1` 格式。
- 导入时应创建不存在的资产；已存在资产应按 `asset_type + market + symbol` 复用。
- 导入通过为每项配置生成一笔 `BUY` 交易来还原当前持仓数量、平均成本和费用，再由系统自动重算持仓。
- 默认跳过已存在活跃持仓，避免重复导入造成重复加仓。
- 导入后应刷新资产配置列表和首页持仓数据，并尝试重算历史盈亏。
- 非法 JSON、错误 schema、非法资产代码、负数数量或成本必须给出清晰错误反馈。

边界说明：

- 资产配置导入导出不是完整数据备份恢复。
- 若用户需要保留完整交易历史，应备份数据库文件或后续使用完整备份恢复功能。
- CSV / Excel 导入导出不属于当前实现，后续可作为持仓列表导出或批量导入增强。

### 5.5 定投补录与待确认净值

系统支持基金定投补录，目标是减少用户逐条录入历史定投交易的成本。

定投补录要求：

- 支持一次录入多期买入交易。
- 支持按金额或份额生成每期交易。
- 已知净值的期次可直接生成正式 `BUY` 交易。
- 净值暂未披露的期次可保存为“待确认”记录。
- 待确认记录应包含份额确认日、净值日期、定投模式、每期金额或份额、费用、备注和标签。
- 后台任务应定期扫描待确认记录，若 Provider 已能返回对应净值，则自动折算补录为正式交易流水。
- 待确认记录应支持后台查看和删除。
- 自动回填失败或过期时，应保留状态或在日志 / 后台反馈中体现。

### 5.6 实时行情与基金净值

行情和基金净值服务通过 Provider Adapter 接入，业务层不直接依赖具体供应商。

股票和场内基金行情数据至少包含：

- 最新价格
- 昨日收盘价
- 前一交易日收盘价
- 今日开盘价
- 最高价
- 最低价
- 成交量
- 涨跌额
- 涨跌幅
- 市场交易状态
- 行情时间戳
- 数据来源标识

场外基金净值数据至少包含：

- 最新单位净值
- 累计净值（若 Provider 可用）
- 前一净值
- 净值日期
- 日涨跌幅
- 基金类型
- 数据来源标识

基金数据规则：

- 场内基金和 ETF 按实时行情处理，计算逻辑与股票一致。
- 场外基金按单位净值处理，通常不具备股票级实时价格。
- 场外基金当日净值未披露时，如 Provider 提供估算净值，可标记为“估算”；否则今日盈亏标记为不可计算或使用最后成功净值并提示。

刷新要求：

- 支持自动刷新和手动刷新。
- 自动刷新间隔可配置，当前允许 5 秒到 600 秒。
- A 股、美股、港股应按各自交易时段展示交易状态。
- 如果行情服务失败，页面不能清空已有数据，应保留最后成功价格并显示异常状态。
- 如果基金净值服务失败，页面不能清空已有净值，应保留最后成功净值并显示异常状态。
- 如果部分资产刷新失败，只标记失败资产，不影响其他资产展示。
- 服务端启动后应执行一次刷新，并在刷新后写入今日历史快照。

### 5.7 盈亏计算

#### 5.7.1 今日盈亏

今日盈亏用于表示当前交易日内持仓相对于上一交易日收盘价、上一净值或当日成交价的浮动变化。

计算公式：

```text
隔夜持仓盈亏 = (最新价或最新净值 - 上一交易日收盘价或上一净值) * 今日开盘前持仓数量
当日买入盈亏 = (最新价或最新净值 - 买入成交价) * 当日买入且仍参与当日收益的数量
当日卖出盈亏 = (卖出成交价 - 该部分持仓的今日收益基准价) * 当日卖出数量
今日盈亏金额 = 隔夜持仓盈亏 + 当日买入盈亏 + 当日卖出盈亏
今日盈亏比例 = 今日盈亏金额 / 今日参与收益的基准金额
```

若当日没有买入或卖出，公式退化为 `(最新价或最新净值 - 上一交易日收盘价或上一净值) * 当前持仓数量`。如果计算所需的上一交易日收盘价、上一净值或当日成交价缺失，则该资产今日盈亏标记为不可计算，并在总览中展示数据缺口提示。

#### 5.7.2 昨日盈亏

昨日盈亏用于复盘上一完整交易日或上一净值披露日的持仓表现。

计算公式：

```text
昨日盈亏金额 = (昨日收盘价或昨日净值 - 前一交易日收盘价或前一净值) * 昨日持仓数量
昨日盈亏比例 = 昨日盈亏金额 / (前一交易日收盘价或前一净值 * 昨日持仓数量)
```

如果用户没有历史持仓快照，MVP 可先使用当前持仓数量近似计算，并在 UI 中标记为估算值或在数据缺口中提示。

#### 5.7.3 总持仓盈亏

总持仓盈亏用于表示当前持仓相对于成本的未实现盈亏。

计算公式：

```text
总持仓盈亏金额 = (最新价或最新净值 - 平均成本价) * 当前持仓数量 - 累计费用
总持仓盈亏比例 = 总持仓盈亏金额 / (平均成本价 * 当前持仓数量 + 累计费用)
```

其中平均成本价与当前持仓数量均由交易流水按加权平均法自动计算。

#### 5.7.4 历史盈亏

历史盈亏支持按以下维度查看：

- 近 7 天
- 近 30 天
- 近 90 天
- 今年
- 自定义日期区间

历史盈亏图表应支持的能力：

- 账户累计盈亏曲线。
- 每日或聚合周期盈亏柱状图。
- 日 / 周 / 月 / 年粒度聚合。
- 鼠标悬停查看具体日期、金额、比例和主要贡献资产。
- 点击或选择走势节点后展示该日或该周的资产贡献。

后续增强能力：

- 按市场拆分的盈亏贡献。
- 按资产类型拆分的盈亏贡献。
- 单只股票或基金的独立历史趋势。

系统支持 A 股、美股、港股和基金资产，市场代码如下：

### 5.12 后台管理密码

系统不提供多用户账号体系。后台管理密码是单实例内的本地保护机制，用于防止误操作或他人直接修改持仓、交易流水和显示设置。

密码设置规则：

- 首次初始化数据库时生成默认后台密码 `admin`。
- 用户进入后台后可以修改后台密码；公网部署前必须修改默认密码。
- 新增资产、录入交易、编辑交易、删除交易、清仓、删除资产、保存显示设置、刷新汇率、校验行情和重算历史等后台写操作必须处于已解锁状态。
- 密码验证通过后可进入短时解锁状态，MVP 默认解锁 30 分钟。
- 修改密码必须先输入旧密码。
- 修改密码后应撤销所有已解锁会话。
- 忘记密码不支持在线找回；MVP 只允许通过本地数据库维护或重新初始化安全设置处理。

安全要求：

- 密码不得明文存储。
- 服务端或本地存储中只保存加盐哈希和密码配置状态。
- 解锁会话 token 只在当前浏览器标签页持有，服务端只保存 token hash。
- 后台写接口需要 `X-Admin-Token`，并校验 `Origin` / `Referer` 来源。
- 密码输入框必须隐藏明文，并支持确认密码一致性校验。
- 连续输错密码应触发临时锁定，避免暴力尝试。
- 解锁时可要求验证码，降低自动化暴力尝试风险。

- 静态资源应使用 ETag、Last-Modified 和合适的 Cache-Control。
- 可压缩静态资源应支持 Brotli / gzip 压缩。
- 刷新行情时不得阻塞页面交互。
- 移动端应能查看核心数据和完成主要后台操作。

## 8. 非功能需求

### 8.1 性能

- 首页首屏加载目标小于 2 秒。
- 1000 条持仓以内，列表筛选、搜索、分页和排序应保持流畅。
- 行情刷新不应阻塞页面交互。
- 图表渲染应支持按需降采样、分页或区间聚合。
- 静态资源应有浏览器缓存策略，构建产物可使用长期缓存。

### 8.2 可用性

- 关键数据必须展示更新时间。
- 盈亏不可计算时必须说明原因。
- 页面刷新或网络短暂中断后，用户已录入但未提交的数据不应无提示丢失。
- 空状态应引导用户添加第一只股票或基金。
- 后台写操作应有保存成功、失败、刷新中、校验中等反馈。
- 移动端应避免表格挤压，优先使用卡片或横向滚动。

### 8.3 安全

- 用户持仓、交易记录和设置属于敏感数据。
- 系统不提供多用户账号、登录注册和角色权限。
- 资产新增、交易录入、交易编辑、交易删除、清仓、删除资产和设置修改必须处于后台解锁状态。
- 后台管理密码不得明文存储，只能保存加盐哈希。
- 后台解锁会话必须有过期时间，服务端不得保存明文 token。
- 默认后台密码为 `admin`，生产或公网部署前必须修改。
- 后台写接口必须校验 `X-Admin-Token`，并校验同源 `Origin` / `Referer`。
- 行情 Provider 密钥必须存储在服务端或安全配置中，不应暴露到前端。
- 公网部署应优先启用 HTTPS，并正确透传 `Host`、`X-Forwarded-Host`、`X-Forwarded-Proto`。

### 8.4 可靠性

- 行情 Provider 失败时，系统应降级到最后成功数据。
- 汇率数据失败时，统一结算货币汇总应提示不可用或使用最后成功汇率。
- 基金净值数据失败时，场外基金应保留最后成功净值并标记数据状态。
- 关键计算逻辑应有自动化测试覆盖。
- 启动时应自动初始化数据库表结构。
- 新库无历史数据时，可回填一段历史快照以保证曲线可用。

## 9. UI 与视觉要求

整体风格：

- 极客、干净、图表化。
- 信息密度高，但布局要清晰。
- 避免营销式大横幅、装饰性卡片堆叠和无意义插画。
- 首屏必须是可操作的盈亏看板。

视觉要求：

- 支持深色、浅色、自动跟随系统和自定义主题。
- 支持自定义背景图、暗度遮罩和模糊程度。
- 正收益和负收益使用清晰且一致的颜色区分，并允许用户选择红涨绿跌或绿涨红跌。
- 图表优先，表格辅助。
- 关键数字使用等宽数字或易读字体。
- 操作按钮使用明确图标和短文本。
- 页面在桌面端优先优化，同时保证移动端可查看核心数据。

交互要求：

- 刷新按钮必须有加载状态。
- 自动刷新时不得导致表格滚动位置丢失。
- 用户筛选条件应在页面内保持。
- 图表和列表之间应尽量支持联动筛选或联动展示。
- 错误、空状态、加载状态必须有明确 UI。
- 后台保存和校验操作应避免按钮文字换行、布局跳动和移动端遮挡。

## 10. 验收标准

### 10.1 功能验收

- 用户可以添加 A 股、美股、港股股票，以及场内基金、ETF 和场外基金。
- 用户通过录入买入 / 卖出交易维护持仓，持仓数量、平均成本与累计费用由系统按加权平均自动计算，不需手动填写平均成本。
- 同一标的再次买入按加仓处理，平均成本自动重算；卖出后平均成本价不变。
- 用户可以批量补录基金定投交易；净值待披露期次可以保存为待确认记录。
- 待确认定投记录在净值可用后可被后台任务自动折算补录为正式交易。
- 用户可以在后台导出当前活跃持仓配置为 JSON 文件。
- 用户可以导入同格式 JSON 文件，系统创建不存在的资产并通过买入交易还原持仓配置；默认跳过已存在活跃持仓。
- 用户可以使用后台密码进入后台；默认密码为 `admin`，进入后可修改密码。
- 新增资产、录入交易、编辑交易、删除交易、清仓、删除资产和保存设置时，系统会要求处于后台解锁状态。
- 后台连续输错密码会临时锁定，解锁可要求验证码。
- 用户可以设置统一结算货币，并在 CNY、USD、HKD 之间切换。
- 用户可以设置结算时区。
- 首页可以展示今日盈亏、昨日盈亏、历史盈亏和总持仓盈亏。
- 首页总览、图表和汇总盈亏会按统一结算货币展示。
- 持仓列表可以按市场、资产类型筛选，按盈亏排序，按股票或基金搜索，并支持分页。
- 实时行情刷新成功时，价格、涨跌幅、交易状态和更新时间同步更新。
- 基金净值刷新成功时，最新净值、净值日期、涨跌幅和更新时间同步更新。
- 行情刷新失败时，页面保留最后成功数据并显示异常状态。
- 历史盈亏图表支持近 7 天、近 30 天、近 90 天、今年和自定义区间。
- 历史盈亏图表支持日、周、月、年粒度聚合。
- 用户可以切换深色 / 浅色 / 自动 / 自定义主题，调整背景图、遮罩、模糊和涨跌配色。

### 10.2 数据验收

- 单实例内不允许重复创建同一 `asset_type + market + symbol` 资产。
- 持仓平均成本与数量按加权平均法从交易流水计算，结果与本文档 5.3 成本规则一致。
- 今日盈亏、昨日盈亏、总持仓盈亏计算公式与本文档一致。
- 后台管理密码只保存加盐哈希，不保存明文。
- 后台解锁 token 只保存哈希，并在过期后失效。
- 统一结算货币汇总必须展示汇率时间和数据来源。
- 结算时区必须是有效 IANA 时区。
- 场外基金当日净值缺失时，今日盈亏必须标记为估算或不可计算。
- 删除或清仓持仓后，历史盈亏数据仍应尽量可追溯。
- 待确认定投记录转为正式交易后，持仓和历史数据应自动重算。

### 10.3 体验验收

- 用户打开首页即可看到核心盈亏，不需要进入二级页面。
- 数据刷新状态明确可见。
- 空状态能引导用户添加股票或基金。
- 图表和列表没有明显遮挡、溢出或文字重叠。
- 移动端可查看总览、持仓卡片，并能完成添加资产、编辑交易、清仓和设置保存等主要操作。
- 页面整体符合极客、干净、图表化的视觉方向。
